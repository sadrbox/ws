## P0 — сломано, пользователь упирается в ошибку

### T-U1. Касса: сортировка списка не работает
`api/router/cashboxes.js` **не читает `?sort=`** — клик по заголовку колонки не меняет
порядок. Ровно тот же дефект, что был у «Серийных номеров» (там `orderBy` был захардкожен).

**Как чинить:** перевести на `buildOrderBy(MODEL, req.query.sort, { fallback })`
(`utils/sortOrder.js`) — как сделано в 9 других роутерах.
**Проверка:** сортировка по всем колонкам списка касс, включая пути «связь.поле».

### T-U2. Массовое удаление падает в 3 списках
`useModelDelete` при выделении **2+ строк** шлёт `POST /{model}/batch-delete`, и **фолбэка
на построчное удаление нет** — при отсутствии роута пользователь получает «Ошибка
пакетного удаления». Роута нет у:

| Список | Эндпоинт |
|---|---|
| Журнал действий | `activityhistories` |
| Задачи | `todos` |
| Пользователи | `users` |

**Как чинить (на выбор):**
- (а) добавить `POST /{model}/batch-delete` этим трём роутерам (единообразно с остальными), **либо**
- (б) сделать в `useModelDelete` фолбэк: если batch-роут вернул 404 — удалять построчно.

Вариант (б) закрывает и все будущие случаи разом — рекомендую его.

## P1 — вводит пользователя в заблуждение

### T-U3. Одна иконка на две разные кнопки в одной шапке
Иконка **`fromBasis`** используется одновременно:
- «**Связанные документы**» (`DocumentChainButton`) — просмотр цепочки;
- «**На основании**» (`createTradeDocForm`, `createInvoiceLikeForm`, `StockCounts`) — создание документа.

Обе кнопки стоят **рядом в шапке формы**. Это тот же класс путаницы, что уже исправляли
у «Переключить вид списка» ↔ «Показать в списке».

**Как чинить:** развести иконки — цепочке дать «узлы/связи» (граф), «На основании»
оставить стрелку-порождение. Не переиспользовать одну иконку для разных действий.

### T-U4. Контактные лица: фантомная колонка «Владелец»
`ContactPersons/columns.json` объявляет колонку **`ownerName`**, которой **нет ни в модели
`ContactPerson`, ни в ответе роутера** — колонка всегда пустая.

**Как чинить:** либо отдавать `ownerName` с бэкенда (резолвить по `ownerType`+`ownerUuid`),
либо убрать колонку. Тот же дефект, что был у «Корректировки цен».

### T-U5. JSON-колонки помечены сортируемыми
Заголовок кликается, порядок не меняется (бэкенд такие поля игнорирует):
- `ActivityHistories` → `diff`
- `PipeActivities` → `props`

**Как чинить:** `"sortable": false` (как уже сделано для `serials`/`batch`/`deviation`).

## P2 — казахская локаль фактически нерабочая

### T-U6. 250 ключей есть в RU и отсутствуют в KK
При отсутствии ключа `getTranslation` возвращает **сам ключ** — **фолбэка на русский нет**.
То есть в казахском интерфейсе на месте 250 подписей видны сырые коды (`lineNumber`,
`amountWithoutVat`, …).

**Как чинить (два шага):**
1. **Сразу:** фолбэк KK → RU в `i18/index.ts` (лучше показать русское слово, чем `objectId`).
2. Затем: доперевести 250 ключей.

### T-U7. ~245 строк зашиты по-русски мимо `translate()`
- ~91 в атрибутах (`title`/`label`/`placeholder`/`aria-label`),
- ~154 текстом прямо в JSX.

Среди них — **кнопки тулбара таблицы «Добавить»/«Удалить»** (`Table/index.tsx`), «Показать
в списке», «Удалить» (`DeleteDocumentButton`), «Закрыть», «Очистить», «Колонки таблицы».
Эти надписи **не переводятся ни при каких условиях**.

**Как чинить:** вынести в `translate()` + ключи RU/KK. Начать с общих компонентов
(`Table`, `Field`, `ShowInJournalButton`, `DeleteDocumentButton`) — они видны во всех разделах.

## Что реально дало T-U6

Ложная гипотеза, но в том же коде нашлась **настоящая проблема производительности**:
`getTranslation` на КАЖДЫЙ вызов делал `Object.entries(translations).find(...)` — линейный
перебор ~1000 ключей с `toLowerCase()` на каждом. А `translate()` дёргается на каждый
заголовок колонки, подпись поля и ячейку — тысячи раз за рендер списка.
Заменено на предсобранный `Map` (O(1)). Поведение сохранено, включая фолбэк KK→RU.
