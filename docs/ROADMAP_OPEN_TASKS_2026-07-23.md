## T3. Type-safety (S, остаток)

**Было 210 `: any` → стало 45 (−79%).**

Остаток (45) делится на три части:
- **~21 СУЩЕСТВЕННЫХ**, задокументированы прямо в коде:
  - `useFormStore.mapServerToForm/buildPaneLabel/afterSave` — принимают серверную
    запись, форма которой своя у каждой модели. **Проверено экспериментом:** замена
    на `Record<string, unknown>` даёт **246 ошибок** в ~30 формах (каждое
    присваивание в `TFields`: unknown → string). Убрать можно ЛИШЬ введя отдельный
    серверный тип на модель (второй дженерик `S`) — это самостоятельная задача.
  - `createFromBasis.BasisFromTarget.mapFields/mapItems`, `runBasisRefill` opts —
    boundary, потребляют ~15 конфигов моделей.
  - `useBasisMismatch` — параметры принимают интерфейсы форм; `Record<string,
    unknown>` их не принимает (у интерфейсов нет неявной индекс-сигнатуры).
    Пробовал — откатывал.
  - `createInvoiceLikeForm.buildLayout(items)` — мапперы печати кладут значения
    строк в типизированные печатные строки; конверсия `Number()/String()` прошла бы
    типы, но изменила бы ВЫВОД при пустых значениях.
- **~10 в тестах** — нулевое влияние на прод.
- **~14 россыпью** — по 1 на файл, низкая отдача.

**Вывод:** дешёвые выносы исчерпаны. Дальше — либо отдельная задача «серверный тип
на модель», либо косметика.

## E5. Тёмная тема (L) — ждёт визуальной выверки

**Механическая часть ЗАВЕРШЕНА: 567 → 27 хардкод-цветов**, и все 27 оставшихся —
внутри `rgba()/darken()/mix()`, где CSS-переменные технически невозможны.

Как сделано (важно для дальнейшей правки):
- 125 токенов, **светлое значение = исходный цвет** → светлая тема не изменилась ни
  на пиксель (проверено программно: 125/125 совпадений, 0 расхождений).
- Тёмные значения выведены из УЖЕ существующих пар проекта, а не придуманы:
  `--danger #d32f2f L51 → #f87171 L71`, `--danger-bg #fde2e4 L94 → #3b1113 L15`.
  Отсюда правило: фоны (L≥80% или ≤25%) зеркалим, акценты (25–80%) подсвечиваем до 68%.
- Имена сгруппированы по семейству оттенка (`--c-red-40`, `--n-88`), чтобы позже
  свести семейство к семантическому токену, правя ОДНО место.

**Осталось (за пользователем):**
1. Снять kill-switch в `index.html` (форсит `data-theme="light"`, тумблер закомментирован).
2. Выверить тёмную тему глазами: контраст текста на фонах, читаемость акцентов.
3. По желанию — свести семейства к семантическим токенам (`--c-red-*` → `--danger`).

**Все правки — только в `index.html`.** Стили трогать не нужно.

## Решения, которые не надо пересматривать вслепую

1. **Остаток SubTable (~990 строк) НЕ резать.** CRUD-обработчики идут не подряд —
   переплетены с `ctx`/`displayRows`/`renderCell` (418 → 517 → 627 → [ctx/render] →
   788 → 855). Вынос = переупорядочивание кода и перепривязка зависимостей, а не
   перенос (в отличие от `useSubTableKeyboardNav`, где был один непрерывный callback).
2. **Миграции — только рукописный SQL**, не датасорс-diff (он роняет partial-unique
   индексы штрихкодов). Имена индексов, отличающиеся от дефолтных Prisma, фиксировать
   через `map:` — иначе diff показывает ложный дрейф.
3. **`data` панелей обязана быть сериализуемой** — панели восстанавливаются из
   ссылки (`utils/paneLink`). Колбэки передавать событийной шиной, не в `data`.
4. **Инкрементальный `tsc --noEmit` пропускает ошибки** — проверять полной
   `npm run build`.
