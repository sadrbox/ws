## T3. Type-safety: убрать источники `any` (M, инкрементально)

**Состояние.** ~407 `any`, источники: `createFromBasis` (34), `useFormStore` (29),
`OrphanRefs` (16), `TradeDocumentItemsTable` (15), `ProductPriceProcessing` (14),
`ProductImportExport` (13). tsc сейчас 0 — `any` глушат каскад, а не чинят.

**Почему не бульдозером.** Часть `any` сущностна: `createFromBasis` — обобщённый
маппинг произвольных строк документов; насильная типизация даёт churn и каскад в
критическом пути (цепочки «На основании»).

**Задачи (best practice — «типизируй ИСТОЧНИК, мерь tsc после каждого модуля»):**
1. Ввести общие типы строк/полей: `type DocRow = Record<string, unknown>`,
   `type DocFields = Record<string, unknown>` в `Table/types`. Не `any`, а
   `unknown` + сужение на месте доступа — компилятор заставит проверять.
2. Начать с `useFormStore` (от него тянется больше всего): типизировать
   `store.getSnapshot().fields`, `setField`, `patch`. Прогон tsc → чинить всплывшее.
3. Затем `createFromBasis`: типизировать API-ответы (`api.get<T>`) и сигнатуры
   `mapFields/mapItems` через `DocRow`, оставив пасс-тру-словари `unknown`.
4. Списочные модели (`OrphanRefs`, `Product*Processing`) — по одной, каждая
   отдельным коммитом с зелёным tsc.

**Критерий готовности:** `grep -c ': any'` падает помодульно; tsc остаётся 0;
ни одного нового `as any` для обхода.

## T4. Декомпозиция Table (2064) и SubTable (1399) (L)

### Table/index.tsx — уже разбит ЛОГИЧЕСКИ, но в одном файле

Внутри 2064 строк живут ~15 обособленных `memo`-компонентов вокруг общего
`TableContext`:

```
TableControlPanel · TableArea · TableFooter · TableHeader · TableBody ·
TableBodyRow · TableConfigModalForm · TableConfigColumns(+Item) ·
DateRangeBar · FieldDateRangeModal · FieldFastSearchInternal
```

Это удача: они уже развязаны через контекст, поэтому декомпозиция — **вынос по
файлам, а не переписывание**. Порядок от листьев к корню (минимизирует правки
импортов):

1. `Table/config/` — `TableConfigModalForm`, `TableConfigColumns`,
   `TableConfigColumnsItem` (модалка настройки колонок, самодостаточна).
2. `Table/toolbar/` — `DateRangeBar`, `FieldDateRangeModal`,
   `FieldFastSearchInternal`, `TableControlPanel`.
3. `Table/parts/` — `TableHeader`, `TableFooter`, `TableArea`, `TableBody`,
   `TableBodyRow`.
4. `TableContext` + типы → `Table/context.ts` (как уже сделано для SubTable).
5. `index.tsx` остаётся оркестратором `Table` + реэкспорт.

**Риск:** `TableContext` несёт много полей — при выносе следить, чтобы каждый
кусок брал из контекста, а не через новые пропы (иначе размажется состояние).
После каждого шага — `tsc` + прогон списков (Sales/Purchases/номенклатура) в
рантайме, а не только сборка.

### SubTable/index.tsx — монолит, сложнее

В отличие от Table, `SubTable` — **один компонент на 1000 строк** (379–1399), а не
набор под-компонентов. Часть уже вынесена: `rowModel.ts` (чистые примитивы),
`useSubTableRows.ts` (стейт-машина строк), `context.ts`. Осталось разгрузить сам
компонент:

1. Выделить `useSubTableColumns` — синхронизация состава колонок (недавняя правка
   `mergeColumnDefs` + пер-товар серии/партии живут прямо в теле).
2. Выделить `useSubTableToolbar` — кнопки панели, inline-add, пересчёт.
3. Ячейки (`ReadOnlyCell` и рендер `renderCell`) → `SubTable/cells.tsx`.
4. Тело таблицы строк → `SubTableBody`.

**Критерий готовности:** `index.tsx` каждого < ~500 строк; поведение проверено на
живых формах (не только tsc); существующие тесты `subTable*.test` зелёные.

**Почему L, а не M:** это ядро, через которое идут ВСЕ документы и списки. Спешка =
регрессии в самом нагруженном месте. Делать по одному под-компоненту за коммит.

## E5. Тёмная тема (L) — поглощает U2-остаток и U4

**Состояние (измерено).** 496 хардкод-цветов в 43 SCSS-модулях; палитра на
**compile-time SCSS-переменных** (`$color3` и т.п.) — в рантайме НЕ переключаются.
Семантические токены цвета уже введены (U2/U4) и заданы для обеих тем — фундамент.

**Почему L.** Настоящая тема = перевод всего цветового слоя на CSS-custom-properties
+ выверка контраста по каждому компоненту в обеих темах. Половинчатая тема выглядит
сломанной.

**Задачи (best practice — token-level theming):**
1. Спроецировать SCSS-палитру в CSS-переменные: сгенерировать `:root { --color3: … }`
   из `variables.scss`, а в модулях заменить `$color3` → `var(--color3)`. Это
   развязывает цвет от сборки, не меняя вид (light-значения = текущие).
2. Определить dark-набор тех же переменных под `@media (prefers-color-scheme: dark)`
   и `:root[data-theme="dark"]` (toggle перекрывает медиа-запрос в обе стороны).
3. Заменить оставшиеся хардкод-hex в модулях на семантические/палитра-токены
   (сюда же уходит U4 — свод серых к шкале).
4. Тумблер темы в шапке (localStorage `theme`), по образцу переключателя языка.
5. Пройти по компонентам с проверкой контраста (focus-ring, disabled, статусы) в
   ОБЕИХ темах — не инвертировать наивно.

**Критерий:** ни одного `#hex` в модулях вне определения токенов; обе темы читаемы;
скриншот-обход ключевых экранов.

## E4. WebSocket / live-статусы (L) — нужны продуктовые решения

**Состояние.** Прикладного реалтайма нет. Есть offline-sync с **периодическим
polling** (`syncManager` → `setInterval`). E4 — это его замена/дополнение push-каналом.

**Сначала решить (продукт):** ЧТО именно должно быть «живым»?
- статусы проведения/обмена (ЭСФ/СНТ ушла → пришёл ответ);
- уведомления другим пользователям организации (создан документ, изменён остаток);
- «кто сейчас смотрит/редактирует документ» (presence)?

**Уточнение (2026-07-21):** E4 = **чат между пользователями в пределах доступных
организаций** (часть collaboration-подсистемы, см. E9), а НЕ односторонние
уведомления. Чат двусторонний — но это не отменяет SSE.

**Почему SSE, а не сразу WebSocket.** Чат = получать + отправлять. Получение —
идеально для SSE (сервер→клиент, авто-реконнект, обычный HTTP, проходит через
cloudflared без апгрейда протокола). Отправка — обычный `POST /messages`. То есть
**SSE(вниз) + POST(вверх) = полноценный чат**, проще WS. WebSocket оправдан только
если нужны: индикатор «печатает…», presence «кто онлайн», мгновенное «прочитано»,
задержка < 100 мс. **Развилка за продуктом:** MVP на SSE+POST либо сразу WS ради
presence/typing.

**Задачи:**
1. **Модель.** `ChatMessage` (uuid, organizationUuid, authorUuid, body, createdAt,
   deletedAt) + `ChatRead` (userUuid, lastReadAt) для непрочитанного. Канал —
   по организации (мультитенант); прямые диалоги 1:1 позже, если понадобятся.
2. **Мультитенант-изоляция.** Сообщения видны только пользователям, у кого есть
   `AccessRight` на эту организацию (тот же `tenantFilter`/`allowedOrgUuids`, что и
   везде). Пользователь пишет в чат каждой доступной ему организации.
3. **Транспорт вниз (SSE).** Express `GET /chat/stream` (SSE), фильтр по доступным
   организациям; шина `EventEmitter` в процессе публикует новое сообщение
   подписчикам его организации. Heartbeat-комментарий каждые ~15 c (иначе cloudflared
   рвёт по таймауту).
4. **Отправка (POST).** `POST /chat/messages` → пишет в БД → публикует в шину.
5. **Клиент.** Хук `useChat(orgUuid)`: SSE-подписка + react-query для истории;
   панель чата в шапке/сайдбаре; бейдж непрочитанного.
6. **Тот же канал — для live-статусов.** Раз SSE-шина уже есть, через неё же слать
   уведомления о событиях учёта (провёлся документ, назначена задача) — переиспользуя
   инфраструктуру, а не строя вторую.
7. **Масштаб.** Несколько инстансов Node → in-process EventEmitter не хватит:
   Postgres `LISTEN/NOTIFY` как шина (без Redis).

## E9. Collaboration: задачи между пользователями + KPI (M)

**Переопределено (2026-07-21):** это НЕ sales-воронка. CRM здесь = **постановка
задач между пользователями в пределах доступных организаций (мультитенант) по
ведению учёта + визуальные показатели эффективности пользователей.**

**Состояние — половина уже построена:**
- `Todo` УЖЕ имеет `curatorUuid` (постановщик) + `executorUuid` (исполнитель) +
  `deadline` + `status`. Есть фронт `models/Todos` (список + форма). То есть
  назначение задач между пользователями смоделировано и работает.
- `ActivityHistory` пишет `userUuid` / `actionType` / `actionDate` /
  `organizationUuid` — **готовый источник KPI**: кто сколько документов создал/
  провёл/удалил по каждой организации.

Значит E9 в этой трактовке — **не новая доменная модель, а UI + агрегация**. Это
**M, а не L**, продуктовых развилок воронки нет.

**Задачи:**
1. **Доска задач (канбан).** Колонки по `Todo.status` (new / in progress / done /
   …), drag&drop между колонками = смена статуса. Мультитенант-фильтр: только
   организации, доступные пользователю (`AccessRight`/`allowedOrgUuids`).
2. **Персональные виды.** «Мои задачи» (`executorUuid = me`), «Поставленные мной»
   (`curatorUuid = me`), «Просроченные» (`deadline < now && status != done`).
   Фильтры поверх существующего `Todos`-эндпоинта.
3. **Уведомление о назначении.** При создании/переназначении `Todo` — событие в
   SSE-шину E4 (исполнителю прилетает бейдж), переиспользуя канал чата.
4. **Дашборд эффективности.** Агрегат `ActivityHistory` по пользователю за период:
   документов создано/проведено, задач закрыто в срок / просрочено. Рендер через
   существующий `ReportPane` (карточки-показатели + простые бары). Изоляция —
   `tenantFilter`.
5. **Статусы задач — справочник, не хардкод-enum.** Разные команды ведут учёт
   по-своему; набор статусов настраиваемый (по образцу `DealStage`, но для Todo).
6. **Права.** `AccessPermission` на модель `Todo` уже работает через общий гард —
   проверить, что доска и дашборд под ним.

**Риск:** не плодить сущности. Задача = существующий `Todo`; активность/аудит =
`ActivityHistory`; коммуникация = чат E4. Новое — только доска, персональные
фильтры и агрегат KPI, а не второй модуль задач.
